# MERRA2 VisIt database plugin

A native VisIt database reader for NASA MERRA-2 3D pressure-level files
(`inst3_3d_asm_Np`, NetCDF-4). It opens the raw `.nc4` directly and presents a
height-corrected rectilinear grid, so VisIt reads the NASA files with **no
Python conversion step**. It is the formatter (`build_merra_series.py`)
reimplemented as a plugin: the pressure → pseudo-height map lives in `GetMesh()`.

## Files

- `MERRA2.xml` — plugin descriptor (database, type MTSD).
- `avtMERRA2FileFormat.h` / `.C` — the reader. This is the part I wrote; the
  rest is generated.
- `MERRA2PluginInfo.*`, `MERRA2CommonPluginInfo.C`, `MERRA2EnginePluginInfo.C`,
  `MERRA2MDServerPluginInfo.C` — boilerplate generated by `xml2plugin`.
- `CMakeLists.txt` — generated by `xml2cmake` (empty until you generate it in a
  dev environment, see below).

## What the reader does

- `OpenFile()` opens the NetCDF, reads the `time`/`lev`/`lat`/`lon` dimensions
  and the axis arrays, and converts pressure to pseudo-height
  `z = 7.5 · ln(1000 / p)`.
- `PopulateDatabaseMetaData()` declares one rectilinear mesh `atmosphere` and
  the scalar fields present (`T`, `QV`, `U`, `V`, `OMEGA`, …).
- `GetMesh()` builds the `vtkRectilinearGrid` (lon, lat, height).
- `GetVar()` reads one variable for one timestep. NetCDF's row-major
  `T(time,lev,lat,lon)` is lon-fastest, which is already VTK's point order, so
  it reads straight into a `vtkFloatArray`.

## Honest status

This source has **not been compiled.** The binary VisIt install used here ships
the code generators (`xml2plugin`, `xml2cmake`) but **not** the development SDK
(no `include/` headers, no libs, no CMake/compiler). You cannot build a plugin
against a plain binary install — you need a VisIt built from source (or a dev
package) plus a C++ toolchain. On Windows this is genuinely hard; plugin builds
are normally done on Linux.

So treat this as a complete, readable plugin you would build in a real VisIt dev
environment, not a `.dll` that drops into this install today.

## How you would build it (in a VisIt dev environment)

```bash
# 1. generate the CMake file (needs VISITARCHHOME, which 'visit -xml2cmake' sets)
visit -xml2cmake MERRA2.xml

# 2. point CMake at your NetCDF, then build
cmake -DCMAKE_BUILD_TYPE=Release \
      -DNETCDF_INCLUDE_DIR=/path/to/netcdf/include \
      -DNETCDF_LIBRARY=/path/to/netcdf/lib/libnetcdf.so .
make

# 3. the build drops libIMERRA2Database / libMMERRA2Database / libEMERRA2Database
#    into your ~/.visit/<version>/<arch>/plugins/databases/, and VisIt then
#    opens MERRA-2 .nc4 files natively.
```

Do **not** re-run `xml2plugin -clobber` after this point — it regenerates the
reader skeleton and overwrites `avtMERRA2FileFormat.C`. Use `xml2cmake` to
refresh only the CMake file.

## Why MTSD and not a writer

VisIt reads MERRA-2 once the plugin exists; there's nothing to write back, so
`haswriter="false"`. One file is 8 timesteps on one grid, so MTSD (Multiple
Timestep, Single Domain) is the right base class.
